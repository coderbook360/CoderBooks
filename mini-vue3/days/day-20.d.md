## 1. 知识图谱回顾

我们构建的响应式系统，可以概括为一张图：

```
        +-----------------+
        |    Reactive     |  <-- Proxy 拦截
        +--------+--------+
                 |
        +--------v--------+
        |  Track / Trigger|  <-- 依赖收集与触发
        +--------+--------+
                 |
        +--------v--------+
        |     Effect      |  <-- 副作用管理
        +--------+--------+
                 |
    +------------+------------+
    |            |            |
+---v---+    +---v---+    +---v---+
|Computed|   | Watch |    | Render|
+-------+    +-------+    +-------+
```

1.  **Reactive/Ref**：数据的守门员，负责拦截读写操作。
2.  **Track/Trigger**：系统的神经中枢，负责建立数据与副作用的联系。
3.  **Effect**：系统的执行单元，负责运行具体的业务逻辑。
4.  **Computed/Watch**：基于 Effect 衍生出的高级功能。
5.  **Scheduler**：系统的调度中心，负责性能优化和执行时机控制。

## 2. 源码对比：理想 vs 现实

我们的实现虽然功能完备，但为了教学清晰，做了一些简化。
让我们看看官方源码里有哪些我们没写的"黑科技"。

### 2.1 位运算标记（Bitwise Markers）

在 Vue 3 源码的 `effect.ts` 中，你会看到大量的位运算：

```typescript
// Vue 3 Source
trackOpBit = 1 << ++effectTrackDepth
if (effectTrackDepth <= maxMarkerBits) {
  initDepMarkers(this)
}
```

**这是做什么的？**
这是为了优化依赖收集的性能。
Vue 3 使用位图（Bitmap）来标记依赖的新旧状态，从而在 `cleanup` 阶段能够更高效地移除无效依赖，而不是像我们那样简单粗暴地全部清空再重新收集。

**为什么我们不写？**
因为这属于极致的性能优化，逻辑非常晦涩，且不影响核心功能的理解。对于学习者来说，理解"双向记录"和"清理机制"的原理比纠结于位运算更重要。

### 2.2 递归检测

在我们的实现中，如果 `effect` 内部修改了它依赖的数据，会导致无限递归。
虽然我们在 `trigger` 中加了 `effect !== activeEffect` 的判断，但 Vue 3 的处理更完善。

Vue 3 维护了一个 `parent` 链，在执行前会检查整个调用栈，确保不会重复执行同一个 `effect`。

### 2.3 数组的极致优化

对于数组，Vue 3 做了非常细致的处理。
比如 `push`、`pop` 等方法，Vue 3 不仅暂停了依赖收集，还对 `length` 属性的依赖做了特殊处理，以应对各种边缘情况。

## 3. 核心思想提炼

通过这两周的实战，你应该掌握了以下核心思想：

1.  **元编程（Metaprogramming）**：利用 `Proxy` 和 `Reflect` 拦截语言的基本操作，这是 Vue 3 响应式的基石。
2.  **发布-订阅模式（Pub-Sub）**：`track` 是订阅，`trigger` 是发布，`targetMap` 是事件中心。
3.  **惰性求值（Lazy Evaluation）**：`computed` 的设计哲学，不到万不得已绝不计算。
4.  **调度机制（Scheduling）**：通过微任务队列实现批量更新，这是现代前端框架性能优化的标配。
