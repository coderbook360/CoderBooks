## 3️⃣ 技术突破点

### 突破1：深入理解 Proxy 机制
**之前的认知**：
- 只知道 Proxy 可以拦截操作
- 不清楚为什么要用 Reflect

**突破后的理解**：
- Proxy 是元编程的体现
- Reflect 保证 this 指向正确
- receiver 参数在继承场景中的关键作用
- WeakMap 防止内存泄漏的原理

**应用场景**：
- 数据验证
- 访问日志
- 响应式系统
- 不可变数据

---

### 突破2：依赖收集的核心算法
**核心数据结构**：
```
targetMap: WeakMap {
  target1 => Map {
    'key1' => Set [effect1, effect2],
    'key2' => Set [effect3]
  },
  target2 => Map { ... }
}
```

**算法流程**：
```
1. 读取属性（get）
   → 记录当前 activeEffect
   → 存入 targetMap[target][key]

2. 设置属性（set）
   → 从 targetMap[target][key] 取出所有 effect
   → 逐个执行

3. 优化点
   → 使用 WeakMap 自动垃圾回收
   → 使用 Set 自动去重
   → 懒代理减少初始化开销
```

**学到的编程思想**：
- 发布订阅模式的实际应用
- 数据结构选择的重要性
- 性能与功能的权衡

---

### 突破3：ref 与 reactive 的设计差异
**理解的转变**：

**之前**：不理解为什么需要 ref，reactive 不是够用了吗？

**之后**：
- ref 解决基本类型响应式问题
- .value 是必需的，用于拦截访问
- 自动解包（unwrap）提升使用体验
- ref 和 reactive 各有适用场景

**设计哲学**：
- API 设计要考虑用户体验
- 类型系统辅助减少错误
- 性能优化不能以牺牲可用性为代价

## 4️⃣ 遇到的挑战与解决

### 挑战1：循环引用导致栈溢出
**问题描述**：
```typescript
const obj = reactive({})
obj.self = obj
console.log(obj.self.self.self...) // 💥 栈溢出
```

**解决过程**：
1. 分析：递归代理导致无限循环
2. 方案1：使用 WeakMap 记录已代理对象 ❌（还是会溢出）
3. 方案2：在 get 中判断返回值是否已是代理 ✅
4. 最终方案：添加 ReactiveFlags 标记

**学到的经验**：
- 边界情况一定要考虑
- 递归代码要有终止条件
- 官方源码的每一行都有原因

---

### 挑战2：computed 的缓存失效时机
**问题描述**：
computed 应该只在依赖变化时重新计算，但我的实现每次都计算。

**解决过程**：
1. 阅读官方源码
2. 发现需要 `dirty` 标志位
3. 在 scheduler 中设置 dirty = true
4. get 时检查 dirty，false 则返回缓存

**核心代码**：
```typescript
class ComputedRefImpl {
  private _value: any
  private _dirty = true
  
  get value() {
    if (this._dirty) {
      this._value = this.effect.run()
      this._dirty = false
    }
    return this._value
  }
}
```

---

### 挑战3：数组方法的特殊处理
**问题描述**：
`arr.push()` 会触发两次 set（length 和索引），导致 effect 重复执行。

**解决方案**：
```typescript
const instrumentations = {
  push(...args) {
    // 暂停依赖收集
    pauseTracking()
    const res = Array.prototype.push.apply(this, args)
    // 恢复依赖收集
    enableTracking()
    return res
  }
}
```

**学到的经验**：
- 数组是特殊的对象，需要特殊处理
- 批量操作需要控制依赖收集
- 官方源码的复杂度是有道理的
