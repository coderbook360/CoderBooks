# 什么是代码架构：前端视角的理解

**思考一个问题**：为什么有些项目越写越顺手，而有些项目越写越痛苦？

答案往往不在于技术栈的选择，而在于**代码架构**。

## 架构的本质是什么

### 从一个比喻开始

想象你要盖一栋房子。你可以不画图纸直接开工——砖头堆在哪里、房间怎么划分，走一步看一步。小木屋可能勉强能成，但如果是二十层高楼呢？

**架构就是软件的"图纸"。**

但软件架构比建筑图纸更复杂，因为软件需要不断变化。用户需求在变，技术在迭代，团队在扩张。好的架构不仅要让当前能工作，还要让未来能演进。

### 架构的三个层次

```
┌─────────────────────────────────────────────────────┐
│                   系统架构                           │
│  微服务划分、前后端分离、数据流向、部署策略             │
├─────────────────────────────────────────────────────┤
│                   应用架构                           │
│  模块划分、状态管理、路由设计、API 层设计              │
├─────────────────────────────────────────────────────┤
│                   代码架构                           │
│  文件组织、组件设计、函数抽象、类型系统                │
└─────────────────────────────────────────────────────┘
```

本书聚焦于**代码架构**——这是每个开发者每天都在面对的决策。

## 前端代码架构的特殊性

**问一个问题**：前端架构和后端架构有什么不同？

### 1. UI 驱动的复杂性

后端代码通常是"请求-处理-响应"的线性流程。前端代码则要处理：

- 用户交互的不确定性（点击、滑动、输入）
- 异步数据流的复杂性（加载、错误、竞态）
- 状态同步的挑战（本地状态、服务器状态、URL 状态）

```typescript
// 一个看似简单的按钮，背后的状态有多复杂？
function SubmitButton() {
  // 状态1：按钮是否禁用
  const [isDisabled, setIsDisabled] = useState(false);
  // 状态2：是否正在提交
  const [isLoading, setIsLoading] = useState(false);
  // 状态3：提交结果
  const [result, setResult] = useState<'success' | 'error' | null>(null);
  // 状态4：错误信息
  const [errorMessage, setErrorMessage] = useState('');
  // 状态5：重试次数
  const [retryCount, setRetryCount] = useState(0);
  
  // 这些状态之间还有复杂的依赖关系...
}
```

### 2. 组件化的双刃剑

组件化是前端的核心范式，但它带来了新的架构挑战：

- **组件边界**：什么逻辑放在组件内，什么抽到外部？
- **组件通信**：父子、兄弟、跨层级组件如何通信？
- **组件复用**：如何设计真正可复用的组件？

### 3. 构建工具的影响

前端代码需要经过打包、编译、压缩。这影响了架构决策：

- 代码分割策略
- 模块加载方式
- 依赖管理

## 架构决策的核心问题

每个架构决策都在回答这些问题：

### 问题 1：如何组织代码

```
# 按类型组织？
src/
├── components/
├── hooks/
├── services/
├── utils/
└── types/

# 还是按功能组织？
src/
├── features/
│   ├── user/
│   │   ├── components/
│   │   ├── hooks/
│   │   └── services/
│   └── order/
│       ├── components/
│       ├── hooks/
│       └── services/
└── shared/
```

**没有"正确"答案，只有"适合"答案。** 关键是理解每种选择的权衡。

### 问题 2：如何管理依赖

```typescript
// 直接导入 — 紧耦合
import { apiClient } from './api-client';

function fetchUser(id: string) {
  return apiClient.get(`/users/${id}`);
}

// 依赖注入 — 松耦合
function createUserService(apiClient: ApiClient) {
  return {
    fetchUser: (id: string) => apiClient.get(`/users/${id}`),
  };
}
```

### 问题 3：如何处理变化

好的架构应该让常见的变化容易应对：

- 添加新功能时，需要修改多少文件？
- 修改一个模块时，会影响多少其他模块？
- 替换一个依赖时，需要改动多少代码？

## 架构思维的培养

### 不是一蹴而就

架构能力来自于：

1. **理解原则**：SOLID、DRY、KISS 等
2. **学习模式**：设计模式、架构模式
3. **实践反思**：每次遇到问题，思考架构层面的原因
4. **阅读源码**：学习优秀项目的架构决策

### 从小处开始

不需要一开始就设计完美的架构。从这些小决策开始训练：

- 这个函数应该放在哪个文件？
- 这个组件应该拆分吗？
- 这个逻辑应该用 Hook 还是普通函数？

## 总结

- **架构是软件的骨架**，决定了代码的可维护性和可扩展性
- **前端架构有其特殊性**：UI 驱动、组件化、构建工具的影响
- **架构决策的核心**：如何组织代码、管理依赖、应对变化
- **架构能力需要持续培养**：学习原则、实践反思、阅读源码

接下来，我们将学习衡量"好代码"的标准，为架构决策提供判断依据。
