# 为什么要手写 jQuery

> "我听过了，我忘记了；我看过了，我记住了；我做过了，我理解了。" —— 孔子

## 一个看似荒谬的问题

"都 2024 年了，React、Vue、Svelte 满天飞，你告诉我要手写 jQuery？"

这个问题我听过无数次。每次我的回答都是：**你完全误解了我们在做什么**。

我们不是在学习一个"过时的库"，我们是在学习**前端开发的底层逻辑**。

让我问你几个问题：

1. React 的 `useState` 是怎么实现的？为什么调用顺序不能变？
2. Vue 的响应式系统是如何追踪依赖的？
3. 链式调用是怎么实现的？为什么 `$('#box').addClass('active').css('color', 'red')` 能一直点下去？

如果这些问题你都能回答得很好，恭喜你，你已经是一个优秀的工程师了。

但如果不能，那说明你一直在"使用"工具，而不是"理解"工具。

## 造轮子的真正价值

我见过两种程序员：

**第一种**：只会调 API。遇到问题就 Google，Stack Overflow 上有答案就照抄，没有就束手无策。框架出了 breaking change，抱怨连天。

**第二种**：理解底层原理。遇到问题能推理出解决方案，能读懂框架源码，甚至能给开源项目提 PR。框架升级了，知道改了什么、为什么改。

区别在哪？**第二种人造过轮子**。

造轮子的价值从来不是那个轮子本身，而是造轮子的过程中，你被迫思考的那些问题：

- 这个功能要怎么设计 API 才好用？
- 这个边界情况要怎么处理？
- 性能瓶颈在哪里？怎么优化？
- 别人是怎么解决同样问题的？

这些问题，在你只是"使用"一个库的时候，永远不会思考。

## 为什么是 jQuery？

市面上有那么多可以造的轮子，为什么偏偏选 jQuery？

### 1. jQuery 是 DOM 操作的集大成者

jQuery 用一套优雅的 API 让繁琐的 DOM 操作变得简洁流畅。

```javascript
// 原生 JavaScript
const els = document.querySelectorAll('.item');
els.forEach(el => {
  el.style.color = 'red';
  el.style.fontSize = '14px';
  el.addEventListener('click', handler);
});

// jQuery
$('.item').css({ color: 'red', fontSize: '14px' }).click(handler);
```

**一行代码解决了选择、遍历、样式设置、事件绑定四个问题。**

这不仅仅是代码量的减少，更是一种 **API 设计的艺术**——如何让复杂的操作变得直观、易用。

### 2. jQuery 的设计模式无处不在

你以为 jQuery 过时了？看看这些"现代"的库：

**Lodash 的链式调用**：
```javascript
_.chain([1, 2, 3])
  .map(n => n * 2)
  .filter(n => n > 2)
  .value()
```

**Axios 的配置合并**：
```javascript
axios.create({
  baseURL: 'https://api.example.com',
  timeout: 1000,
  headers: { 'X-Custom-Header': 'foobar' }
})
```

**D3.js 的选择器和链式操作**：
```javascript
d3.selectAll('p')
  .style('color', 'blue')
  .attr('class', 'highlight')
```

这些设计模式的源头，都可以追溯到 jQuery。

**链式调用、配置合并、选择器引擎、事件委托、插件机制**——这些 jQuery 在 2006 年就玩明白的东西，今天依然是库设计的基石。

### 3. jQuery 的代码量适中，适合深入学习

React 的源码有几万行，读起来需要大量前置知识。Vue 3 的响应式系统虽然精巧，但涉及 Proxy、WeakMap 等较新的语言特性。

而 jQuery 的核心代码只有不到 6000 行（不含 Sizzle），涉及的都是最基础的 JavaScript 知识：**原型链、闘包、this 指向、DOM API**。

这意味着，只要你有扎实的 JS 基础，就能读懂、写出 jQuery 的核心功能。

## 你将收获什么

通过手写 Mini-jQuery，你将掌握：

### 1. 设计模式的实战应用

不是背书式地"工厂模式是什么"，而是亲手体验"为什么这里要用工厂模式"：

- **工厂模式**：`$()` 函数为什么不需要 `new`？
- **外观模式**：如何用一个简单的 API 封装复杂的底层操作？
- **迭代器模式**：`$.each()` 是怎么实现的？
- **观察者模式**：事件系统的本质是什么？
- **命令模式**：动画队列是怎么管理的？

### 2. JavaScript 核心机制的深入理解

你会真正理解这些"玄学"问题：

- 原型链到底是怎么回事？
- `this` 指向为什么这么让人困惑？
- 闭包在实际开发中怎么用？
- 为什么要用 `call`、`apply`、`bind`？

### 3. DOM 操作的底层逻辑

不再是"我知道用 `querySelector`"，而是：

- 浏览器是怎么解析 CSS 选择器的？
- 为什么 `innerHTML` 有时候很慢？
- 事件委托为什么能提高性能？
- `offsetWidth` 和 `clientWidth` 有什么区别？

### 4. 工程化思维

不再是写一个能跑的 demo，而是：

- 如何设计可扩展的 API？
- 如何处理边界情况？
- 如何写可测试的代码？
- 如何发布一个 npm 包？

## 这本书的学习方式

我们不会一上来就讲一个"完整"的 jQuery 实现。那样太无聊了，也学不到东西。

我们会采用**渐进式**的方式：

1. **提出问题**：这个功能要解决什么痛点？
2. **最简实现**：用最少的代码实现核心功能
3. **发现问题**：这个简单实现有什么缺陷？
4. **迭代优化**：一步步完善，处理边界情况
5. **最终方案**：展示生产级的实现

每一步都有可运行的代码，每一个设计决策都有充分的解释。

## 开始之前

你需要准备：

- Chrome 浏览器（本书所有示例均基于最新版 Chrome 测试）
- 一个代码编辑器（VS Code 推荐）
- Node.js 环境（用于后续的打包和发布）
- 对 JavaScript 基础的扎实理解

如果你对原型链、闭包、this 还不太熟悉，建议先补一补基础。这本书假设你已经是一个有一定经验的 JavaScript 开发者。

> **注意**：本书只考虑最新版 Chrome 浏览器，不涉及任何旧版浏览器兼容性问题。这让我们可以专注于设计模式和实现技巧本身，而不是被兼容性代码分散注意力。
