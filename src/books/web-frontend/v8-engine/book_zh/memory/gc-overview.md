# 垃圾回收概览：V8 的内存管理架构

你有没有想过，为什么 JavaScript 不需要手动管理内存，却能高效运行？为什么有时候程序会突然卡顿，然后又恢复正常？答案就藏在 V8 的垃圾回收（Garbage Collection，GC）机制中。

本章将带你建立对 V8 垃圾回收的整体认知，理解为什么 V8 需要多种回收算法，以及它们是如何协同工作的。

## 为什么需要垃圾回收？

让我们从一个真实的内存泄漏问题开始：

```javascript
// 这段代码会不断消耗内存，最终导致程序崩溃
function memoryLeak() {
  const cache = [];
  
  setInterval(() => {
    // 每秒添加 1MB 数据到缓存
    const data = new Array(1024 * 1024).fill(0);
    cache.push(data);
    
    console.log(`缓存大小: ${cache.length}MB`);
  }, 1000);
}

memoryLeak();
// 1 秒后：缓存大小: 1MB
// 2 秒后：缓存大小: 2MB
// 3 秒后：缓存大小: 3MB
// ...
// 几分钟后：内存溢出，程序崩溃
```

在 C/C++ 这样的语言中，你必须手动调用 `free()` 或 `delete` 来释放内存。但 JavaScript 作为高级语言，将这个负担交给了**垃圾回收器**。

垃圾回收器的核心任务是：**识别并回收那些不再被使用的对象，释放它们占用的内存。**

## V8 的堆内存结构

在深入回收算法之前，我们需要理解 V8 如何组织内存。

V8 的堆（Heap）被划分为几个区域，其中最重要的是**新生代**和**老生代**：

```
V8 堆内存布局：

┌────────────────────────────────────────────────────────────┐
│                       V8 Heap                              │
├────────────────────────────────────────────────────────────┤
│                                                            │
│  ┌──────────────────────────────────────────────┐         │
│  │        新生代 (Young Generation)               │ 很小   │
│  │  ┌──────────────┬──────────────┐              │ (几MB) │
│  │  │ From Space   │  To Space    │              │         │
│  │  │  (活跃区)     │  (空闲区)     │              │         │
│  │  └──────────────┴──────────────┘              │         │
│  └──────────────────────────────────────────────┘         │
│                                                            │
│  ┌──────────────────────────────────────────────┐         │
│  │        老生代 (Old Generation)                 │ 很大   │
│  │                                              │ (几百MB) │
│  │  大部分对象最终会被晋升到这里                   │         │
│  └──────────────────────────────────────────────┘         │
│                                                            │
│  ┌──────────────────────────────────────────────┐         │
│  │        大对象空间 (Large Object Space)          │         │
│  │  超过某个阈值的对象直接分配在这里               │         │
│  └──────────────────────────────────────────────┘         │
│                                                            │
│  ┌──────────────────────────────────────────────┐         │
│  │        代码空间 (Code Space)                   │         │
│  │  存储 JIT 编译后的机器码                       │         │
│  └──────────────────────────────────────────────┘         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 为什么要分代？

这是基于**分代假说（Generational Hypothesis）**：
- **大部分对象的生命周期都很短**（如临时变量、函数作用域内的对象）
- **少部分对象会存活很长时间**（如全局对象、缓存数据）

基于这个假说，V8 采用了**分代回收策略**：
- **新生代**：频繁、快速的回收，处理短命对象
- **老生代**：低频、完整的回收，处理长命对象

## V8 的垃圾回收算法

V8 使用多种回收算法，针对不同代采用不同策略。

### 1. Scavenge 算法（新生代）

**特点**：
- **速度快**：牺牲空间换时间
- **频率高**：新生代对象分配快，很快会满
- **适合场景**：短命对象的快速回收

**核心思想**：
```
1. 将新生代分为两个相等的空间：From Space 和 To Space
2. 新对象分配在 From Space
3. 当 From Space 满时触发 GC：
   - 扫描 From Space 中的存活对象
   - 将存活对象复制到 To Space
   - 清空 From Space
   - 交换 From 和 To 的角色
```

**示意图**：
```
回收前（From Space 已满）：
┌─────────────┬─────────────┐
│ From Space  │  To Space   │
│ ┌─┐ ┌─┐ ┌─┐│             │
│ │A│ │B│ │C││  (空闲)      │
│ └─┘ └─┘ └─┘│             │
│  ↑   X   ↑ │             │
│ 存活 死亡 存活│             │
└─────────────┴─────────────┘

回收后（存活对象复制到 To Space）：
┌─────────────┬─────────────┐
│ From Space  │  To Space   │
│             │ ┌─┐     ┌─┐ │
│  (清空)      │ │A│     │C│ │
│             │ └─┘     └─┘ │
│             │  ↑       ↑  │
│             │  存活    存活 │
└─────────────┴─────────────┘

下一轮分配时，From 和 To 交换角色
```

**代价**：
- ⚠️ 空间利用率只有 50%（To Space 始终空闲）
- ⚠️ 需要复制对象，大对象复制成本高

### 2. Mark-Sweep-Compact 算法（老生代）

**特点**：
- **完整回收**：处理所有老生代对象
- **频率低**：只在老生代空间不足时触发
- **适合场景**：长命对象的完整回收

**核心思想**（三个阶段）：
```
1. Mark（标记）：标记所有存活对象
   - 从 GC Roots 开始遍历（全局对象、栈上的引用）
   - 递归标记所有可达对象

2. Sweep（清除）：清除未标记的死亡对象
   - 遍历堆，回收未标记的对象

3. Compact（整理）：整理内存碎片
   - 将存活对象移动到内存的一端
   - 消除内存碎片，为新对象分配提供连续空间
```

**示意图**：
```
标记阶段：
根对象 → 对象A ← 对象B → 对象C
         (标记)   (死亡)   (标记)

清除阶段：
对象A     [空]     对象C
(保留)   (回收)   (保留)

整理阶段：
对象A  对象C  [大片连续空闲空间]
```

**代价**：
- ⚠️ 标记和整理阶段需要遍历所有对象，耗时长
- ⚠️ 整理阶段需要移动对象，更新所有引用

### 3. 增量标记（Incremental Marking）

**为什么需要增量标记？**

Mark-Sweep-Compact 需要暂停 JavaScript 执行（Stop-The-World），老生代越大，暂停时间越长。对于用户来说，就是页面卡顿。

**解决方案**：**将标记工作分散到多个小步骤中**，每个步骤执行完后恢复 JavaScript 执行，避免长时间暂停。

```
传统标记（Stop-The-World）：
JS执行 → [长时间GC暂停] → JS执行
         ^^^^^^^^
         页面卡顿

增量标记（Incremental）：
JS执行 → [小GC] → JS执行 → [小GC] → JS执行 → [小GC] → ...
         ^^^^      ^^^^      ^^^^
         用户几乎感觉不到
```

### 4. 并发标记（Concurrent Marking）

**更进一步的优化**：**在后台线程中进行标记**，完全不阻塞主线程。

```
主线程：  JS执行 → JS执行 → JS执行 → ...
          ↓        ↓        ↓
后台线程：[标记] → [标记] → [标记] → ...
```

## V8 的完整 GC 流程

现在我们理解了各种算法，让我们看看 V8 是如何协同使用它们的：

```javascript
// 模拟对象的生命周期
function simulateObjectLifecycle() {
  // 1. 新对象在新生代分配
  const temp = { data: 'temporary' };
  
  // 2. 新生代满了，触发 Scavenge
  // temp 如果仍被引用，会被复制到 To Space
  
  // 3. temp 经历了多次 Scavenge 仍然存活
  // → 晋升到老生代
  
  // 4. 老生代空间不足
  // → 触发 Mark-Sweep-Compact
  
  // 5. temp 不再被引用
  // → 在老生代 GC 中被回收
}
```

完整的回收流程：

```
1. 新对象分配在新生代的 From Space

2. From Space 满 → 触发 Scavenge
   - 复制存活对象到 To Space
   - 清空 From Space
   - 交换 From 和 To

3. 对象在新生代经历 2 次 Scavenge 仍存活
   → 晋升到老生代

4. 老生代空间不足 → 触发 Mark-Sweep
   - 增量标记阶段（分多个小步骤）
   - 并发标记阶段（后台线程）
   - 清除阶段
   - 整理阶段（如果内存碎片过多）

5. 重复 1-4，持续回收
```

## GC 对性能的影响

垃圾回收虽然自动，但并非没有代价。

### GC 暂停（GC Pause）

**主要影响**：
- Scavenge：暂停 1-5ms（新生代小，影响小）
- Mark-Sweep：暂停 50-100ms（老生代大，影响明显）
- 增量标记：分散暂停，每次 5-10ms

### 内存分配速度

**影响因素**：
- 新生代分配极快（指针移动）
- 老生代分配较慢（需要找合适的空闲空间）
- 碎片化严重时分配更慢

### 优化建议

理解 GC 机制后，我们可以编写对 GC 友好的代码：

1. **减少临时对象创建**
```javascript
// ❌ 不好：频繁创建临时对象
function bad() {
  for (let i = 0; i < 1000000; i++) {
    const temp = { value: i }; // 每次循环创建新对象
    process(temp);
  }
}

// ✅ 好：复用对象
function good() {
  const reused = { value: 0 };
  for (let i = 0; i < 1000000; i++) {
    reused.value = i; // 复用同一个对象
    process(reused);
  }
}
```

2. **避免内存泄漏**
```javascript
// ❌ 不好：事件监听器泄漏
element.addEventListener('click', function handler() {
  // 忘记移除监听器，element 和 handler 都无法回收
});

// ✅ 好：及时清理
const handler = () => { /* ... */ };
element.addEventListener('click', handler);

// 不再需要时
element.removeEventListener('click', handler);
```

3. **合理使用缓存**
```javascript
// ❌ 不好：无限增长的缓存
const cache = {};
function getData(key) {
  if (!cache[key]) {
    cache[key] = expensiveOperation(key);
  }
  return cache[key];
}

// ✅ 好：使用 WeakMap 或设置缓存上限
const cache = new WeakMap();
// 或者使用 LRU 缓存，限制大小
```

## 本章小结

1. **分代假说**：大部分对象生命周期短，少部分对象长命，这是分代回收的理论基础

2. **Scavenge 算法**：用于新生代，牺牲空间换速度，适合短命对象的快速回收

3. **Mark-Sweep-Compact**：用于老生代，完整回收，包含标记、清除、整理三个阶段

4. **增量与并发**：通过增量标记和并发标记，减少 GC 暂停对用户体验的影响

5. **对象晋升**：新生代对象经历多次 Scavenge 仍存活，会晋升到老生代

理解这些核心概念后，你将能够：
- 预测代码的内存行为
- 识别潜在的内存泄漏
- 编写对 GC 友好的高性能代码
