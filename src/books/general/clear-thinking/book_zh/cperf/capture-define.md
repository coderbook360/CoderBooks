# 识别真正的问题

> "一个定义良好的问题，已经解决了一半。" —— 查尔斯·凯特林

## 问题背后的问题

我们经常解决的是症状，而不是根本问题。

### 案例：服务器响应慢

**表面问题**：用户投诉系统慢

**直接原因**：服务器响应时间过长

**深层问题**：
- 为什么响应慢？→ 数据库查询慢
- 为什么查询慢？→ 没有索引
- 为什么没有索引？→ 设计时没考虑
- 为什么没考虑？→ 没有性能设计规范
- 为什么没有规范？→ 团队缺乏这方面意识

**根本问题**：团队缺乏性能设计的意识和规范

如果只解决"加索引"，同样的问题会在其他地方重复出现。

### 问题层次模型

```
          ┌─────────────┐
          │    症状     │  ← 可见的问题表现
          └──────┬──────┘
                 │
          ┌──────┴──────┐
          │  直接原因   │  ← 导致症状的直接因素
          └──────┬──────┘
                 │
          ┌──────┴──────┐
          │  深层原因   │  ← 导致直接原因的因素
          └──────┬──────┘
                 │
          ┌──────┴──────┐
          │  根本原因   │  ← 需要解决的核心问题
          └─────────────┘
```

## 5W2H问题定义法

使用5W2H框架确保问题定义完整：

### What - 是什么

- 具体发生了什么？
- 问题的表现是什么？
- 期望的状态是什么？
- 现状与期望的差距是什么？

### Who - 谁

- 谁遇到了这个问题？
- 谁是利益相关方？
- 谁应该负责解决？
- 谁能提供帮助？

### When - 何时

- 问题什么时候发生的？
- 发生的频率如何？
- 有时间规律吗？
- 解决的截止时间是什么？

### Where - 何处

- 在哪里发生的？
- 影响范围有多大？
- 有地点或环境规律吗？

### Why - 为什么

- 为什么会发生？
- 为什么现在发生？
- 为什么在这里发生？
- 深层原因是什么？

### How - 如何

- 问题是如何表现的？
- 之前是如何处理的？
- 可能的解决方案有哪些？

### How Much - 多少

- 问题的严重程度？
- 影响的规模有多大？
- 解决需要多少资源？
- 不解决会造成多大损失？

## 区分症状与病因

### 症状vs病因检验

问自己：
1. 如果解决了这个问题，根本问题是否消失？
2. 这个问题会不会在其他地方再次出现？
3. 这是短期缓解还是长期解决？

### 5Why分析法

连续追问5个"为什么"：

**问题**：项目延期

1. **为什么**延期？→ 需求变更太多
2. **为什么**需求变更多？→ 初期需求不清晰
3. **为什么**需求不清晰？→ 与客户沟通不充分
4. **为什么**沟通不充分？→ 没有需求确认流程
5. **为什么**没有流程？→ 项目管理规范缺失

**根本问题**：缺乏项目管理规范

### 因果链验证

画出因果链，验证每个环节：

```
项目管理规范缺失
        ↓ 导致
需求确认流程缺失
        ↓ 导致
客户沟通不充分
        ↓ 导致
初期需求不清晰
        ↓ 导致
频繁需求变更
        ↓ 导致
项目延期
```

验证：如果建立项目管理规范，是否能防止延期？

## 问题陈述的逻辑结构

### 好的问题陈述包含：

1. **现状描述**：客观描述当前情况
2. **期望状态**：明确目标
3. **差距量化**：具体的差距是什么
4. **约束条件**：有什么限制
5. **成功标准**：如何判断问题已解决

### 问题陈述模板

```
【现状】：目前的情况是_____________

【期望】：期望达到的状态是_____________

【差距】：现状与期望的具体差距是_____________

【约束】：需要考虑的限制条件包括_____________

【标准】：问题解决的成功标准是_____________
```

### 案例对比

**差的问题陈述**：
"系统性能不好，需要优化。"

**好的问题陈述**：
"【现状】首页加载时间平均5秒，高峰期达到10秒
【期望】首页加载时间控制在2秒以内
【差距】需要将加载时间缩短60%-80%
【约束】不能大改架构，优化周期2周，成本控制在5万以内
【标准】连续一周，95%的请求加载时间<2秒"

## 【检查清单】问题定义完备性检验

使用以下清单验证问题定义的质量：

### 完整性检查

- [ ] 问题的具体表现描述清楚了吗？
- [ ] 5W2H各要素都考虑了吗？
- [ ] 期望状态明确了吗？
- [ ] 差距被量化了吗？

### 准确性检查

- [ ] 这是根本问题还是表面症状？
- [ ] 用5Why分析追溯过了吗？
- [ ] 因果链经过验证了吗？
- [ ] 解决这个问题能达到目标吗？

### 可操作性检查

- [ ] 问题定义得足够具体，可以开始解决吗？
- [ ] 成功标准是可衡量的吗？
- [ ] 约束条件都明确了吗？
- [ ] 问题范围是可控的吗？

### 相关方检查

- [ ] 关键利益相关方认同这个问题定义吗？
- [ ] 问题的优先级与各方一致吗？
- [ ] 资源投入与问题重要性匹配吗？

## 本章小结

- 要解决根本问题，而不是表面症状
- 使用5W2H确保问题定义完整
- 用5Why追溯到根本原因
- 好的问题陈述包含现状、期望、差距、约束和标准
- 使用检查清单验证问题定义质量
