# 确定性决策

> "决策的质量不取决于结果，而取决于过程。"

## 决策的逻辑结构

### 决策的基本要素

一个完整的决策包含：

```
┌─────────────────────────────────────────────────────────┐
│                      决策要素                            │
├─────────────────────────────────────────────────────────┤
│  目标：我想达成什么？                                     │
│  选项：有哪些可选方案？                                   │
│  标准：用什么标准评判？                                   │
│  信息：每个选项在各标准下表现如何？                        │
│  权重：各标准的重要性如何？                               │
│  决定：综合评估后选择哪个？                               │
└─────────────────────────────────────────────────────────┘
```

### 决策流程

```
明确目标
    ↓
生成选项（至少3个）
    ↓
确定评估标准
    ↓
收集信息
    ↓
评估各选项
    ↓
做出选择
    ↓
制定行动计划
```

## 利弊分析与加权评估

### 简单利弊分析

最简单的决策工具——列出优缺点：

```
方案：引入微服务架构

优点（+）                     缺点（-）
─────────────────────────────────────────────
+ 服务独立部署               - 运维复杂度增加
+ 技术栈灵活                 - 网络延迟增加
+ 故障隔离                   - 事务处理复杂
+ 团队独立开发               - 需要新技能
+ 易于扩展                   - 初期投入大
```

### 加权利弊分析

给每个因素加上权重和评分：

| 因素 | 权重 | 方案A得分 | 方案B得分 |
|-----|------|---------|---------|
| 开发效率 | 30% | 8 | 6 |
| 运维成本 | 25% | 5 | 8 |
| 扩展性 | 20% | 9 | 6 |
| 团队接受度 | 15% | 7 | 9 |
| 风险可控 | 10% | 6 | 8 |
| **加权总分** | 100% | **7.15** | **7.15** |

加权总分 = Σ(权重 × 得分)

## 决策矩阵的使用

### 构建决策矩阵

**步骤1：列出选项**
列出所有可选方案（建议3-5个）

**步骤2：确定标准**
确定评估标准（建议5-8个）

**步骤3：设定权重**
根据重要性分配权重（总和100%）

**步骤4：评分**
每个选项在每个标准下打分（1-10分）

**步骤5：计算**
加权求和，比较结果

### 决策矩阵示例

**场景**：选择前端框架

| 标准 | 权重 | React | Vue | Angular |
|-----|------|-------|-----|---------|
| 学习成本 | 20% | 6 | 8 | 5 |
| 生态系统 | 25% | 9 | 8 | 8 |
| 性能 | 20% | 8 | 8 | 7 |
| 团队熟悉度 | 20% | 5 | 8 | 4 |
| 长期维护 | 15% | 8 | 7 | 9 |
| **总分** | 100% | **7.15** | **7.8** | **6.55** |

结论：基于当前标准，Vue是最优选择。

### 决策矩阵的注意事项

1. **选项要真正可行**：不要列入明显不可行的选项
2. **标准要相互独立**：避免重复计算同一因素
3. **权重要反映真实优先级**：不是"政治正确"的权重
4. **评分要基于事实**：不是感觉
5. **结果是参考**：分数接近时需要综合判断

## 识别隐含假设

### 什么是隐含假设

隐含假设是我们在决策中默认为真，但没有明说的前提。

**例子**：
决策："我们应该快速发布，先上线再迭代"

隐含假设：
- 用户能容忍早期的不完善
- 我们能快速响应反馈
- 快速失败的成本可接受
- 市场时机比质量更重要

### 为什么要识别隐含假设

```
┌─────────────────────────────────────────────────────┐
│ 隐含假设错误导致的问题：                               │
│                                                     │
│ 假设：用户能容忍不完善                                │
│ 现实：用户对质量要求很高，给差评后很难挽回             │
│ 结果：口碑受损，用户流失                              │
│                                                     │
│ 如果早识别这个假设，可能会选择不同的策略               │
└─────────────────────────────────────────────────────┘
```

### 如何识别隐含假设

**问题清单**：
- 这个决策要成功，什么必须是真的？
- 我们认为什么是理所当然的？
- 如果环境变化，这个决策还成立吗？
- 反对者会质疑我们的什么假设？

**假设类型**：
- 市场假设：用户需要这个功能
- 技术假设：这个方案技术可行
- 资源假设：我们有足够的人力
- 时间假设：能在deadline前完成
- 成本假设：成本在预算内

### 验证隐含假设

1. **列出假设**：把所有隐含假设写出来
2. **评估风险**：哪些假设如果错了影响最大
3. **设计验证**：如何验证这些假设
4. **及早验证**：在投入大量资源前验证

## 【检查清单】决策逻辑有效性检验

### 目标检验

- [ ] 目标是否清晰明确？
- [ ] 是否区分了必须达成和希望达成？
- [ ] 目标是否与更高层目标一致？
- [ ] 成功的标准是什么？

### 选项检验

- [ ] 是否考虑了足够多的选项？
- [ ] 选项是否相互独立？
- [ ] 是否包含了"不行动"选项？
- [ ] 是否有创造性的选项？

### 标准检验

- [ ] 评估标准是否完整？
- [ ] 标准之间是否独立？
- [ ] 权重是否反映真实优先级？
- [ ] 是否考虑了长期影响？

### 评估检验

- [ ] 评估是否基于事实？
- [ ] 是否考虑了不确定性？
- [ ] 是否识别了隐含假设？
- [ ] 关键假设是否验证？

### 决策检验

- [ ] 决策是否与评估结果一致？
- [ ] 是否考虑了实施可行性？
- [ ] 是否有应急计划？
- [ ] 决策是否可逆？

## 本章小结

- 决策包含目标、选项、标准、信息、权重、决定六要素
- 决策矩阵是系统化评估的有力工具
- 识别隐含假设是减少决策风险的关键
- 使用检查清单确保决策过程完整
