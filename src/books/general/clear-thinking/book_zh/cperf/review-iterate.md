# 检验与迭代

> "第一个想法通常是错的，第二个想法可能更错，第五个想法才开始靠谱。"

## 为什么需要检验

### 思考的盲点

**我们总是高估自己的判断**：
- 确认偏误让我们只看到支持自己的证据
- 过度自信让我们忽视不确定性
- 事后诸葛亮让我们觉得一切都在预料之中

**检验的意义**：
- 发现逻辑漏洞
- 补充遗漏因素
- 修正错误假设
- 提升决策质量

### 检验的时机

```
          决策前
             │
     ┌───────┴───────┐
     ▼               ▼
  快决策          慢决策
  (可逆)          (不可逆)
     │               │
     ▼               ▼
  简单检验       深度检验
  - 直觉检查     - 红队思维
  - 快速推演     - 预演验尸
     │               │
     ▼               ▼
      └───────┬───────┘
             │
          执行中
             │
             ▼
         持续监测
         - 关键指标
         - 预警信号
             │
             ▼
          决策后
             │
             ▼
          复盘学习
          - 对比预期
          - 提取教训
```

## 逻辑检验

### 检查论证结构

**步骤**：
1. 提取核心结论
2. 列出所有前提假设
3. 检验每个前提的可靠性
4. 检验推理过程是否有效

### 常见逻辑漏洞

| 漏洞类型 | 表现 | 检验问题 |
|---------|------|---------|
| 跳跃推理 | 从A直接跳到C | 中间步骤是什么？ |
| 循环论证 | 用结论证明结论 | 论据独立于结论吗？ |
| 虚假二分 | 非此即彼 | 真的只有两个选择吗？ |
| 滑坡谬误 | A必然导致Z | 每一步都必然吗？ |
| 以偏概全 | 从个案推全局 | 样本有代表性吗？ |

### 对立面思考

**问自己**：
- 如果这个结论是错的，会是什么原因？
- 有什么证据可以推翻这个结论？
- 持相反观点的聪明人会怎么说？

## 预演验尸法（Pre-mortem）

### 什么是预演验尸

**传统事后验尸**：项目失败后分析原因
**预演验尸**：假设项目已经失败，回溯可能的原因

### 操作步骤

```
第一步：时间旅行
"假设6个月后，这个项目彻底失败了"
         │
         ▼
第二步：头脑风暴
"它是怎么失败的？列出所有可能的原因"
         │
         ▼
第三步：识别盲点
"哪些原因是我们之前没考虑到的？"
         │
         ▼
第四步：预防措施
"我们现在能做什么来防止这些问题？"
```

### 示例

**项目**：新功能上线

**假设失败后的原因**：
- 用户不理解新功能的使用方式
- 性能问题导致加载太慢
- 和现有功能的冲突引发bug
- 产品经理中途改需求导致延期
- 关键开发人员离职

**预防措施**：
- 增加新手引导
- 提前做性能测试
- 加强回归测试
- 需求冻结协议
- 知识分享降低单点依赖

## 红队思维

### 什么是红队

**来源**：军事演习中扮演敌方的团队
**目的**：专门挑战和攻击己方方案

### 建立内心红队

**步骤**：
1. 完成你的方案
2. 切换角色，假设你要推翻这个方案
3. 穷尽所有攻击点
4. 切换回来，加固方案

### 红队问题清单

**可行性攻击**：
- 技术上真的能实现吗？
- 时间表现实吗？
- 资源足够吗？

**假设攻击**：
- 关键假设可靠吗？
- 如果假设不成立会怎样？
- 有没有黑天鹅风险？

**竞争攻击**：
- 竞争对手会如何反应？
- 他们有什么我们没有的优势？
- 他们可能怎样破坏我们的计划？

**执行攻击**：
- 执行中最可能出问题的是什么？
- 团队真的有动力吗？
- 沟通和协调会出问题吗？

## 迭代思维

### 思考也需要迭代

```
版本1.0 ─────────────────────────────────────→
    │
    ↓ 检验发现问题
    │
版本2.0 ─────────────────────────────────────→
    │
    ↓ 获得新信息
    │
版本3.0 ─────────────────────────────────────→
    │
    ↓ 实践反馈
    │
版本4.0 ─────────────────────────────────────→
```

### 何时迭代

**触发迭代的信号**：
- 发现了新的重要信息
- 关键假设被证明错误
- 实际结果偏离预期
- 环境发生重大变化
- 执行中遇到意外障碍

### 避免过度迭代

**迭代的成本**：
- 决策延迟
- 执行混乱
- 团队疲劳

**平衡原则**：
- 区分核心假设和次要细节
- 设定迭代的触发条件
- 不为小问题改变大方向

- [ ] 准备了备选方案吗？
- [ ] 保持了开放心态吗？

## 本章小结

- 检验是清晰思考不可或缺的环节
- 逻辑检验发现推理漏洞
- 预演验尸法预见潜在失败
- 红队思维从对手角度攻击方案
- 保持迭代思维，随时准备调整
