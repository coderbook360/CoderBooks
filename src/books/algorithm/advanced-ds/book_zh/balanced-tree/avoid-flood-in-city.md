# 避免洪水泛滥

本章我们来解决一道非常有趣的贪心问题——**避免洪水泛滥**。这道题完美展示了有序集合在贪心算法中的应用：我们需要高效地找到"最优的"可用资源。

## 问题描述

**LeetCode 1488. 避免洪水泛滥 (Avoid Flood in The City)**

你的国家有无数个湖泊，所有湖泊一开始都是空的。当第 `n` 个湖泊下雨的时候，如果第 `n` 个湖泊是空的，那么它就会装满水。如果第 `n` 个湖泊已经满了，这就会发生洪水。

给你一个整数数组 `rains`，其中：
- `rains[i] > 0` 表示第 `i` 天会在第 `rains[i]` 个湖泊下雨
- `rains[i] == 0` 表示第 `i` 天是晴天，你可以选择**一个**湖泊并抽干这个湖泊的水

返回一个数组 `ans`：
- 如果 `rains[i] > 0`，`ans[i]` 应该是 `-1`
- 如果 `rains[i] == 0`，`ans[i]` 是你选择抽干的湖泊编号
- 如果无法避免洪水，返回空数组

**示例 1**：
```
输入: rains = [1, 2, 3, 4]
输出: [-1, -1, -1, -1]
解释: 每个湖泊只下一次雨，不需要抽水
```

**示例 2**：
```
输入: rains = [1, 2, 0, 0, 2, 1]
输出: [-1, -1, 2, 1, -1, -1]
解释: 
第 3 天抽干湖泊 2（为第 5 天的雨做准备）
第 4 天抽干湖泊 1（为第 6 天的雨做准备）
```

**示例 3**：
```
输入: rains = [1, 2, 0, 1, 2]
输出: []
解释: 
第 4 天湖泊 1 已满，第 3 天只能抽一个湖泊
无论抽哪个，都会有一个湖泊洪水
```

**提示**：
- 1 <= rains.length <= 10^5
- 0 <= rains[i] <= 10^9

## 思路分析

### 核心问题

当某个湖泊第二次下雨时，我们必须在两次下雨之间找一个晴天来抽干它。

**关键约束**：晴天是有限资源，需要合理分配。

### 贪心策略

当湖泊 `L` 在第 `d1` 天和第 `d2` 天各下一次雨（`d1 < d2`）时：
- 我们必须在 `(d1, d2)` 范围内找一个晴天
- 应该选择**最早的**满足条件的晴天

为什么选最早的？
- 如果选晚的，早的晴天可能对其他湖泊没用（因为时间窗口不匹配）
- 选最早的可以为后续留下更多选择空间

这就是**贪心**：在满足当前约束的前提下，选择对未来影响最小的选项。

## 解法：贪心 + 有序集合

```python
from sortedcontainers import SortedList

class Solution:
    def avoidFlood(self, rains: list[int]) -> list[int]:
        n = len(rains)
        result = [-1] * n
        
        # 记录可用的晴天
        sunny_days = SortedList()
        
        # 记录每个湖泊上次下雨的日期
        last_rain = {}
        
        for day in range(n):
            lake = rains[day]
            
            if lake == 0:
                # 晴天，先记录下来
                sunny_days.add(day)
                result[day] = 1  # 临时值，如果没被使用就抽湖泊 1
            else:
                # 雨天
                if lake in last_rain:
                    # 这个湖泊之前下过雨，需要抽水
                    prev_day = last_rain[lake]
                    
                    # 找第一个 > prev_day 的晴天
                    idx = sunny_days.bisect_right(prev_day)
                    
                    if idx == len(sunny_days):
                        # 没有合适的晴天，无解
                        return []
                    
                    # 使用这个晴天
                    dry_day = sunny_days[idx]
                    result[dry_day] = lake
                    sunny_days.remove(dry_day)
                
                # 更新这个湖泊的上次下雨日期
                last_rain[lake] = day
        
        return result
```

## 执行过程演示

以 `rains = [1, 2, 0, 0, 2, 1]` 为例：

| day | rains[day] | 操作 | sunny_days | last_rain | result |
|-----|------------|------|------------|-----------|--------|
| 0 | 1 | 湖泊1下雨 | [] | {1:0} | [-1,_,_,_,_,_] |
| 1 | 2 | 湖泊2下雨 | [] | {1:0,2:1} | [-1,-1,_,_,_,_] |
| 2 | 0 | 晴天 | [2] | {1:0,2:1} | [-1,-1,1,_,_,_] |
| 3 | 0 | 晴天 | [2,3] | {1:0,2:1} | [-1,-1,1,1,_,_] |
| 4 | 2 | 湖泊2再次下雨 | [3] | {1:0,2:4} | [-1,-1,2,1,_,_] |
| 5 | 1 | 湖泊1再次下雨 | [] | {1:5,2:4} | [-1,-1,2,1,-1,-1] |

**详细分析**：

**Day 4**：湖泊 2 再次下雨
- 上次下雨是 day 1
- 需要在 (1, 4) 范围内找晴天
- `bisect_right(1)` = 0，`sunny_days[0]` = 2
- 使用 day 2 抽干湖泊 2
- `result[2] = 2`

**Day 5**：湖泊 1 再次下雨
- 上次下雨是 day 0
- 需要在 (0, 5) 范围内找晴天
- `bisect_right(0)` = 0，`sunny_days[0]` = 3
- 使用 day 3 抽干湖泊 1
- `result[3] = 1`

## 为什么是 `bisect_right`？

我们需要找**严格大于** `prev_day` 的最小晴天。

- `bisect_left(prev_day)`：找 >= prev_day 的位置
- `bisect_right(prev_day)`：找 > prev_day 的位置

由于晴天必须在 `prev_day` **之后**，我们用 `bisect_right`。

## 复杂度分析

**时间复杂度**：O(n log n)
- 每个晴天最多被添加和删除各一次，每次 O(log n)
- 每个雨天最多触发一次 `bisect_right`，O(log n)

**空间复杂度**：O(n)
- `sunny_days` 最多存储 n 个晴天
- `last_rain` 最多存储 n 个湖泊

## 无解情况分析

什么时候无解？

1. **时间窗口内没有晴天**：两次下雨之间没有晴天
2. **晴天不够用**：虽然有晴天，但被其他湖泊占用了

贪心策略能保证：如果有解，我们一定能找到；如果贪心失败，则确实无解。

**证明**：假设贪心失败（没找到合适晴天），但存在另一个有效分配。那么在那个分配中，必定有某个晴天 `d` 被分配给了某个湖泊 `L'`，而 `d` 也满足当前湖泊的条件。但由于我们选的是最早的满足条件的晴天，`d` 应该已经被我们选中。矛盾。

## 常见错误

### 错误一：忘记处理晴天的默认值

```python
# 错误：result 初始化为 -1，但晴天应该填正数
result = [-1] * n

# 正确：晴天临时填 1，表示抽干湖泊 1（任意正数都行）
if lake == 0:
    sunny_days.add(day)
    result[day] = 1
```

### 错误二：使用 `bisect_left` 而非 `bisect_right`

```python
# 错误：可能找到等于 prev_day 的位置
idx = sunny_days.bisect_left(prev_day)

# 正确：需要严格大于
idx = sunny_days.bisect_right(prev_day)
```

### 错误三：忘记更新 `last_rain`

```python
# 错误：只在第二次下雨时更新
if lake in last_rain:
    # ... 处理第二次下雨
    last_rain[lake] = day  # 有时忘记这一行
else:
    last_rain[lake] = day

# 正确：无论如何都要更新
last_rain[lake] = day
```

## 变体问题

### 变体一：每天可以抽 k 个湖泊

如果每个晴天可以抽干 k 个湖泊，只需修改资源管理逻辑：

```python
sunny_resources = defaultdict(int)  # day -> 剩余可用次数

if lake == 0:
    sunny_resources[day] = k
```

### 变体二：返回任意一个有效解

原题已经是返回任意有效解。如果要返回字典序最小的解，需要更复杂的处理。

## 相关问题

这道题的思路可以推广到其他"资源分配"问题：

- **任务调度**：有 n 个任务和 m 个时间槽，每个任务有时间窗口约束
- **会议室分配**：为会议分配可用的时间段
- **CPU 调度**：为进程分配 CPU 时间片

核心模式都是：**贪心地选择最早/最晚/最优的可用资源**。

## 本章小结

避免洪水泛滥完美展示了有序集合在贪心算法中的应用。

**核心要点**：

1. **贪心策略**：选择最早满足条件的晴天
2. **有序集合的作用**：高效地找到"第一个 > x 的元素"
3. **`bisect_right` vs `bisect_left`**：根据是否需要严格大于选择
4. **资源分配模式**：晴天是资源，需要按需分配

**举一反三**：
- 需要在满足约束的候选中快速找最优 → 有序集合 + 贪心
- 资源有时效性（过期无效）→ 动态维护有序集合
- 需要判断有无解 → 贪心能力边界分析
