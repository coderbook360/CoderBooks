# 实战：分发饼干

> LeetCode 455. 分发饼干 | 难度：简单

贪心分配问题入门，也是理解"一对一匹配"类贪心问题的最佳起点。

📎 [LeetCode 455. 分发饼干](https://leetcode.cn/problems/assign-cookies/)

---

## 题目描述

有一群孩子和一堆饼干。第 i 个孩子的胃口值是 `g[i]`，第 j 块饼干的尺寸是 `s[j]`。

只有 `s[j] >= g[i]`，饼干 j 才能满足孩子 i。每个孩子最多只能给一块饼干。

求最多能满足多少孩子。

**示例**：
```
输入：g = [1,2,3], s = [1,1]
输出：1
解释：只有胃口为1的孩子能被满足

输入：g = [1,2], s = [1,2,3]
输出：2
解释：两个孩子都能被满足
```

---

## 思路分析

### 问题的本质

这是一个**一对一匹配问题**：每个孩子最多匹配一块饼干，每块饼干最多分给一个孩子。

**关键问题**：如何分配才能满足最多的孩子？

### 直觉思考

假设有孩子胃口 `[1, 3]`，饼干尺寸 `[2, 3]`：

- 方案A：用尺寸2的饼干给胃口1的孩子，用尺寸3的给胃口3的 → 满足2个
- 方案B：用尺寸3的饼干给胃口1的孩子 → 浪费！只能满足1个

**洞察**：用"刚好够"的饼干满足孩子，不要浪费大饼干。

### 贪心策略

**策略**：对于每个孩子，用能满足他的最小饼干。

**实现步骤**：
1. 将孩子胃口升序排列
2. 将饼干尺寸升序排列
3. 用双指针匹配：尝试用当前最小饼干满足当前最小胃口的孩子

---

## 代码实现

```typescript
function findContentChildren(g: number[], s: number[]): number {
  // 排序是贪心的前提
  g.sort((a, b) => a - b);
  s.sort((a, b) => a - b);
  
  let child = 0;   // 指向当前待满足的孩子
  let cookie = 0;  // 指向当前尝试的饼干
  
  while (child < g.length && cookie < s.length) {
    if (s[cookie] >= g[child]) {
      // 饼干能满足当前孩子
      child++;  // 满足了一个孩子，移动到下一个
    }
    // 无论是否满足，饼干都"用掉"了（要么分配了，要么太小不够）
    cookie++;
  }
  
  return child;  // child 就是满足的孩子数量
}
```

### 代码逻辑详解

**为什么 `cookie` 每次都要自增？**

- 如果饼干满足了孩子：饼干被分配，不能再用
- 如果饼干太小：这块饼干对后面更大胃口的孩子更没用，直接跳过

---

## 执行过程详解

```
孩子胃口: [1, 2, 3]  (已排序)
饼干尺寸: [1, 1]     (已排序)

第1轮 (child=0, cookie=0):
  饼干尺寸 s[0]=1 >= 孩子胃口 g[0]=1 ✓
  → 满足了第1个孩子
  → child=1, cookie=1

第2轮 (child=1, cookie=1):
  饼干尺寸 s[1]=1 < 孩子胃口 g[1]=2 ✗
  → 饼干太小，跳过
  → cookie=2

cookie=2 >= s.length，循环结束

结果：满足 1 个孩子
```

---

## 贪心正确性证明

### 反证法

假设存在更优解，其中某块大饼干 B 分给了小胃口孩子 A，而小饼干 b 分给了大胃口孩子 C（或没人）。

```
假设：
  大饼干 B → 小胃口 A
  小饼干 b 未分配或给了大胃口 C

调整：
  小饼干 b → 小胃口 A (因为 b >= A 的胃口)
  大饼干 B → 大胃口 C (更可能满足)

调整后满足的孩子数 >= 原方案
```

所以，用小饼干满足小胃口永远不会更差。

### 贪心选择性质

每一步的局部最优（用能满足当前孩子的最小饼干）能导致全局最优。

---

## 复杂度分析

- **时间复杂度**：O(n log n + m log m)
  - 排序孩子：O(n log n)
  - 排序饼干：O(m log m)
  - 双指针遍历：O(n + m)

- **空间复杂度**：O(log n + log m)，排序的递归栈空间

## 另一种贪心：先满足大胃口

从另一个角度思考：优先满足大胃口的孩子。

```typescript
function findContentChildren(g: number[], s: number[]): number {
  g.sort((a, b) => b - a);  // 孩子胃口降序
  s.sort((a, b) => b - a);  // 饼干尺寸降序
  
  let child = 0;
  let cookie = 0;
  
  while (child < g.length && cookie < s.length) {
    if (s[cookie] >= g[child]) {
      // 最大的饼干能满足最大胃口的孩子
      child++;
    }
    cookie++;
  }
  
  return child;
}
```

**两种策略等价**：无论从小到大还是从大到小，最终满足的孩子数相同。

## 常见错误

### 错误1：忘记排序

```typescript
// ❌ 错误：没有排序
function findContentChildren(g: number[], s: number[]): number {
  let count = 0;
  for (let i = 0; i < g.length; i++) {
    for (let j = 0; j < s.length; j++) {
      if (s[j] >= g[i]) {
        count++;
        s[j] = -1;  // 标记已使用
        break;
      }
    }
  }
  return count;
}
```

这种暴力方法可能得到次优解，因为匹配顺序不对。

### 错误2：饼干没用完就停止

```typescript
// ❌ 错误：应该继续尝试
if (s[cookie] < g[child]) {
  break;  // 不应该 break
}
```

当前饼干不够，但后面可能还有更大的饼干（已排序情况下不会发生，但理解错误很常见）。

## 总结

分发饼干问题是贪心算法的入门经典：

1. **排序是贪心的前提**：让我们能按顺序做出最优选择
2. **局部最优推全局最优**：每次用最小够用的饼干，不浪费资源
3. **双指针高效匹配**：排序后线性扫描即可完成匹配

掌握这道题，你就理解了一对一匹配问题的贪心本质。

**任务分配**：
- 工人有能力值，任务有难度值
- 分配最多任务

**背包问题变体**：
- 物品有体积，背包有容量
- 装最多物品

关键：**一对一匹配 + 排序贪心**
