# 11. Error Handling: Building a Robust Parsing and Matching Mechanism

As we near the end of our journey in implementing `mini-path-to-regexp`, let's discuss a topic that is crucial but often relegated to a lower priority during feature development: error handling. A robust library not only works correctly on the "happy path" but also provides clear, predictable behavior and meaningful error messages when faced with various invalid inputs and exceptional situations. This chapter will build an elegant error handling mechanism for our library.

## 11.1. Error Classification: When Do Errors Occur?

Throughout the lifecycle of `path-to-regexp`, errors mainly occur in two stages:

1.  **Parse-time Errors**: When a user provides a syntactically incorrect path string, the `parse` function should fail immediately and throw an error. This is the most common type of error.
2.  **Run-time Errors**: When the `toPath` function generated by `compile` is called, if the passed `data` object does not meet the requirements of the path pattern (e.g., a required parameter is missing), an error should be thrown.

It is worth noting that the `match` function does not throw an error when a match fails; instead, it returns `false`. This is an important design decision because it treats a "no match" scenario as a normal, expected program branch, rather than an "error" that needs to interrupt the program.

## 11.2. Parse-time Error Handling

Parse-time errors are primarily the responsibility of the `parse` function to capture. We need to add validation logic to our implementation of `parse`.

### 11.2.1. Unclosed Parentheses

A common path syntax error is when parentheses are not correctly closed, for example, `/user/:id(\d+`. In the official implementation of `path-to-regexp`, the `parse` function maintains a counter to track the nesting level of parentheses. In our simplified version, we can add a simple check at the end of the `parse` function to catch the most common unclosed cases.

A more robust method is to handle this within `PATH_REGEXP`. The `PATH_REGEXP` we used before is already powerful, but for more precise error capturing, the official implementation actually uses a more complex regular expression and performs parsing in a loop. In our `mini-path-to-regexp`, we can simulate this check by checking for any unhandled `(` characters in the path string at the end of the `parse` function.

```typescript
// Simplified check in parse function
// ... after while loop
if (pathOffset < path.length) {
  // Check for unclosed parentheses in the remaining path
  if (path.substr(pathOffset).includes("(")) {
    throw new TypeError(`Unclosed group in path "${path}"`);
  }
  tokens.push(path.substr(pathOffset));
}
```

### 11.2.2. Illegal Escaping

The `parse` function should also handle illegal escape sequences. For example, if the path is `/a\`, the backslash is at the end of the string and does not escape any character. This should also be treated as a syntax error.

## 11.3. Run-time Error Handling

Run-time errors mainly occur in the `toPath` function generated by `compile`.

```typescript
// In tokensToFunction's returned toPath function
function toPath(data: object = {}): string {
  let path = "";
  for (const token of tokens) {
    if (typeof token === "string") {
      path += token;
      continue;
    }

    const value = data[token.name];

    if (value == null) {
      if (token.modifier === "?" || token.modifier === "*") {
        continue;
      }
      // Key: Throw an error for required parameters
      throw new TypeError(`Expected "${token.name}" to be defined for path "${originalPath}"`);
    }
    
    // ... (Array handling)

    path += token.prefix + encodeURIComponent(value);
  }
  return path;
}
```

Our implementation in Chapter 9 already includes this logic. The core here is:

- When the value of a parameter is `null` or `undefined`, check its `modifier`.
- If it is optional (`?` or `*`), skip it safely.
- If it is required (`modifier` is `""` or `+`), a `TypeError` must be thrown.

The error message thrown should be as friendly and specific as possible, for example, `Expected "id" to be defined`, rather than a vague "parameter error." This greatly helps developers to quickly locate the problem.

## 11.4. Practice: Using `try...catch`

As users of the library, we should always wrap calls to `path-to-regexp` (especially `compile` and `toPath`) in `try...catch` blocks to gracefully handle potential errors and prevent the program from crashing.

```javascript
import { compile } from "./mini-path-to-regexp";

// Catch parse-time errors
try {
  compile("/user/:id("); // Unclosed group
} catch (e) {
  console.error(`[Parse Error]: ${e.message}`);
}

// Catch run-time errors
const toUserPath = compile("/user/:id");
try {
  toUserPath({}); // Missing required parameter 'id'
} catch (e) {
  console.error(`[Runtime Error]: ${e.message}`);
}
```
