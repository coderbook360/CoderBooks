# 10. Advanced Encapsulation: A Detailed Look at the `match` Function Implementation

We have already implemented `pathToRegexp` (for matching) and `compile` (for generating paths) separately. However, in daily use, directly manipulating the regular expression returned by `pathToRegexp` and extracting parameters from it is still a bit cumbersome. Developers need to manually call `regexp.exec()` and then match the values in the result array with the `keys` array to get a structured parameter object.

To provide a more developer-friendly experience, `path-to-regexp` offers a higher-level encapsulation—the `match` function. It packages the process of matching and parameter extraction into a single function call, greatly simplifying its use.

## 10.1. The Signature and Responsibilities of the `match` Function

- **Input**: `path: string`, the path pattern to be matched.
- **Output**: `(pathname: string) => MatchResult | false`, a "matching function." We'll call it `matchFn`.

The `match` function itself also returns a function. This `matchFn` accepts a specific URL `pathname`. If it matches, it returns a `MatchResult` object containing detailed match information; if it fails to match, it returns `false`.

A `MatchResult` object typically includes:

- `path`: The full path that was matched.
- `index`: The starting position of the match.
- `params`: A key-value object containing all the extracted dynamic parameters.

```typescript
// Pseudocode
function match(path, options) {
  const tokens = parse(path);
  const regexp = tokensToRegexp(tokens, undefined, options);

  // Implementation of matchFn
  return function matchFn(pathname) {
    const m = regexp.exec(pathname);

    if (!m) {
      return false;
    }

    // ... construct the params object ...

    return { path: m[0], index: m.index, params };
  };
}
```

## 10.2. Implementation of `matchFn`

The core of `matchFn` is to call the regular expression generated by our implemented `tokensToRegexp` and process the result of `exec` to convert it into a friendly `params` object.

To construct the `params` object, `matchFn` needs to know the parameter name corresponding to each capturing group. This means that `matchFn` needs access to the `tokens` array generated by the `parse` function (or more accurately, the `Key` objects within the `tokens` array). Therefore, the `match` function needs to first call `parse` and pass the `tokens` to the `matchFn` closure.

```typescript
import { parse, tokensToRegexp, Key } from "./internal";

interface MatchResult {
  path: string;
  index: number;
  params: Record<string, any>;
}

function match(path: string, options = {}) {
  const keys: Key[] = [];
  const regexp = tokensToRegexp(parse(path), keys, options);

  return function matchFn(pathname: string): MatchResult | false {
    const m = regexp.exec(pathname);

    if (!m) {
      return false;
    }

    const { 0: path, index } = m;
    const params: Record<string, any> = {};

    for (let i = 1; i < m.length; i++) {
      const key = keys[i - 1];
      const value = m[i];

      if (value !== undefined) {
        if (key.modifier === "*" || key.modifier === "+") {
          params[key.name] = value.split(key.prefix).map(decodeURIComponent);
        } else {
          params[key.name] = decodeURIComponent(value);
        }
      }
    }

    return { path, index, params };
  };
}
```

Let's break down the implementation of `matchFn`:

1.  **Get the `keys` Array**: At the top level of the `match` function, we define a `keys` array and pass it as an "output parameter" to `tokensToRegexp`. This way, after `tokensToRegexp` is executed, the `keys` array will contain the `Key` objects of all dynamic parameters in the path pattern, in the exact same order as the capturing groups in the regular expression.

2.  **Perform the Match**: Inside `matchFn`, we first execute `regexp.exec(pathname)`. If it returns `null` (`!m`), it means there is no match, and we return `false` directly.

3.  **Initialize the `params` Object**: If the match is successful, we destructure the first element (the fully matched `path`) and the `index` from the match result `m`. Then, we initialize an empty `params` object.

4.  **Iterate and Populate `params`**: This is the most crucial step. We start a loop from `i = 1` to iterate through the match result array `m` (since `m[0]` is the full match, capturing groups start from `m[1]`).
    - For each matched value `m[i]`, we get its corresponding `Key` object `keys[i - 1]` from the `keys` array.
    - We check if `value` is `undefined`. If an optional parameter did not appear, the value of its capturing group will be `undefined`, and we simply ignore it.
    - **Handle Repeated Parameters**: If `key.modifier` is `*` or `+`, it means this parameter can have multiple values. The regex in `path-to-regexp` captures these values through repeated capturing groups, but the result of `exec` will only retain the last capture. A more complete implementation would handle this, but a simplified version of `path-to-regexp` might concatenate multiple values into a string separated by a delimiter (`key.prefix`). We then use `split` to restore it to an array and apply `decodeURIComponent` to each element.
    - **Handle Single-Value Parameters**: For regular parameters, we directly decode the `value` with `decodeURIComponent` and then store it in the `params` object with `key.name` as the key.

5.  **Return the Result**: Finally, we return the `MatchResult` object containing `path`, `index`, and `params`.

## 10.3. The "Dual Identity" of the `match` Function

The `match` function in `path-to-regexp` has another very clever design. The `matchFn` function it returns also has a `path` property, which is the `toPath` function generated by the `compile` function!

```typescript
// ... in match function ...
const toPath = tokensToFunction(tokens);
const matchFn = function(...) { ... };
matchFn.path = toPath;
return matchFn;
```

This means that when you call `const matcher = match("/user/:id")`, the `matcher` you get can not only be used to match paths (`matcher("/user/123")`) but also to generate paths (`matcher.path({ id: 123 })`)!

This design elegantly aggregates the two highly related operations of "matching" and "generating" into the same return value, providing great convenience for developers. It perfectly embodies the ultimate pursuit of `path-to-regexp` as a utility library in balancing functionality and ease of use.

By implementing the `match` function, we have brought the journey of implementing the core features of `path-to-regexp` to a successful conclusion. We started from the lowest-level `parse` and `tokensToRegexp`, built up step by step, and finally encapsulated a powerful, easy-to-use, and elegantly designed high-level API.
