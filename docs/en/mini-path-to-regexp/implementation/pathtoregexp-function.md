# 7. Code Generation: Implementing the `pathToRegexp` Function

In the previous chapter, we successfully converted a path string into a structured `Token` array. Now, we enter the second stage of the `path-to-regexp` core workflow: code generation. The goal of this chapter is to implement the `pathToRegexp` function, which will consume the `Token` array and "compile" it into a final, executable JavaScript `RegExp` object.

The `pathToRegexp` function is actually a wrapper function. Its core logic resides in an internal function that we'll call `tokensToRegexp`. The job of `pathToRegexp` is to call `parse` and then pass its result to `tokensToRegexp`.

```typescript
// Pseudocode
function pathToRegexp(path, keys?, options?) {
  const tokens = parse(path);
  return tokensToRegexp(tokens, keys, options);
}
```

Therefore, the focus of this chapter is the implementation of the `tokensToRegexp` function.

## 7.1. Responsibilities of the `tokensToRegexp` Function

- **Input**: `tokens: Token[]`, `keys?: Key[]` (an optional output parameter), `options?: Options` (a configuration object).
- **Output**: A `RegExp` object.

Its core task is to iterate through the `tokens` array, translate each `Token` into a small piece of a regular expression string based on its type and properties, and finally join all the pieces together to create a `RegExp` instance based on the provided `options`.

```typescript
import { Token, Key } from "./types";

interface Options {
  sensitive?: boolean; // Case-sensitive matching
  strict?: boolean;    // Strict matching of the end
  end?: boolean;       // Match to the end of the string (default: true)
  delimiter?: string;  // Default parameter delimiter
}

function tokensToRegexp(tokens: Token[], keys?: Key[], options: Options = {}): RegExp {
  const { sensitive = false, strict = false, end = true, delimiter = "/" } = options;
  let route = "";

  for (const token of tokens) {
    if (typeof token === "string") {
      // Handle static path Tokens
      route += escapeString(token);
    } else {
      // Handle dynamic parameter Key Tokens
      const prefix = escapeString(token.prefix);
      const suffix = escapeString(token.suffix || "");

      if (keys) {
        keys.push(token);
      }

      if (token.pattern) {
        route += `${prefix}(${token.pattern})${suffix}${token.modifier}`;
      } else {
        route += `${prefix}([^${escapeString(delimiter)}]+?)${suffix}${token.modifier}`;
      }
    }
  }

  // ... Handle the end of the path and create the RegExp object ...

  return new RegExp(route, sensitive ? "" : "i");
}
```

## 7.2. Iteration and Translation

The core of `tokensToRegexp` is a `for...of` loop that iterates through the `tokens` array generated by the `parse` function.

### Handling Static Path `Token`s (`string`)

When a `token` is a string, it represents a static path segment. We need to safely append it to the final `route` regular expression string. It's crucial to note that static paths may contain characters with special meaning in regular expressions, such as `.`, `+`, `*`, etc. Therefore, we must use a helper function `escapeString` to escape these characters.

```typescript
function escapeString(str: string): string {
  return str.replace(/[\\^$.*+?()[\]{}|]/g, "\\$&");
}

// ... in tokensToRegexp ...
if (typeof token === "string") {
  route += escapeString(token);
}
```

### Handling Dynamic Parameter `Token`s (`Key`)

When a `token` is a `Key` object, the process is slightly more complex. We need to construct a regular expression segment based on the properties of the `Key` object.

1.  **Populate the `keys` Array**: If the caller provides a `keys` array, we need to push the current `Key` object into it. This allows the caller of `pathToRegexp` to get detailed information about all the parameters.

2.  **Handle Prefix and Suffix**: Like static paths, the `prefix` and `suffix` also need to be escaped.

3.  **Construct the Capturing Group**: This is the most critical step. We need to create a capturing group `(...)` for the parameter.
    - If `token.pattern` exists (i.e., the user provided a custom pattern like `\d+`), we use this `pattern` directly as the content of the capturing group.
    - If `token.pattern` does not exist, we use a default pattern. The default pattern is `[^<delimiter>]+?`, where `<delimiter>` is the separator from `options` (defaulting to `/`). `[^/]+?` means "match one or more non-slash characters, in a non-greedy way."

4.  **Append the Modifier**: Finally, we append the `token.modifier` (`?`, `*`, `+`, or `""`) directly after the capturing group.

The complete `else` branch is as follows:

```typescript
// ... in tokensToRegexp ...
else {
  const prefix = escapeString(token.prefix || "");
  const suffix = escapeString(token.suffix || "");

  if (keys) {
    keys.push(token);
  }

  if (token.pattern) {
    route += `${prefix}(${token.pattern})${suffix}${token.modifier}`;
  } else {
    // Use the default pattern
    route += `${prefix}([^${escapeString(delimiter)}]+?)${suffix}${token.modifier}`;
  }
}
```

## 7.3. Handling the End of the Path

After the loop finishes, the `route` string is mostly complete. However, we still need to handle the end of the path based on the `options`, which determines the strictness of the route matching.

```typescript
// ... after for loop ...
if (end) {
  if (!strict) route += `(?:${escapeString(delimiter)})?`; // Optional trailing slash
  route = `^${route}$`; // Match from start to end
} else {
  // If not matching to the end, handle optional slash for sub-paths
  const endToken = tokens[tokens.length - 1];
  const isEndDelimited = typeof endToken === "string" ? endToken.endsWith(delimiter) : endToken === undefined;
  if (!strict) {
    route += `(?:${escapeString(delimiter)}(?=${route}))?`;
  }
  if (!isEndDelimited) {
    route += `(?=${escapeString(delimiter)}|$)`;
  }
  route = `^${route}`;
}
```

- **`end: true` (default)**: This is the most common case, where we want the path to match completely.
    - `strict: false` (default): We allow an optional trailing slash. For example, both `/user` and `/user/` will match the `/user` pattern. This is achieved by adding `(?:/)?`.
    - `strict: true`: We do not allow an optional trailing slash, requiring an exact match.
    - Finally, we wrap the entire `route` with `^` and `$` to enforce a full match from start to end.

- **`end: false`**: This allows us to match a prefix of a path. For example, using the `/user` pattern to match `/user/123`. The logic here is more complex, using positive lookaheads `(?=...)` to ensure that the path is followed by either a delimiter or the end of the string, without consuming these characters. In our `mini-path-to-regexp`, we can simplify or ignore this part for now to focus on the core functionality.

## 7.4. Creating the `RegExp` Object

The final step is to create the regular expression object using `new RegExp()`. The `flags` parameter is determined by `options.sensitive` to decide whether to add the `i` (ignore case) flag.

```typescript
return new RegExp(route, sensitive ? "" : "i");
```

With this, the `tokensToRegexp` function has fulfilled its mission. Like a translator, it accurately translates a structured `Token` array into a powerful `RegExp` object, preparing everything for the subsequent path matching.
