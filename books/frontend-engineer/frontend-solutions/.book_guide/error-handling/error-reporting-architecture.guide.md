## 3. 内容要点

### 核心概念

| 概念 | 解释要求 |
|-----|---------|
| **错误上报 (Error Reporting)** | 解释将前端错误发送到服务端的完整流程 |
| **错误聚合 (Error Aggregation)** | 解释将相似错误合并，避免重复告警 |
| **SourceMap 还原** | 解释将压缩代码的错误堆栈还原为源码位置 |
| **采样策略** | 解释为何需要采样，如何设计采样比例 |
| **上报时机** | 解释立即上报 vs 批量上报的权衡 |

### 关键知识点

1. **错误上报系统架构**
   - 采集层：捕获各类错误
   - 处理层：格式化与丰富上下文
   - 传输层：上报策略与通道
   - 存储层：错误数据存储
   - 分析层：聚合、告警与可视化

2. **错误数据采集**
   - 错误类型与分类
   - 错误上下文信息
   - 用户环境信息
   - 行为轨迹信息

3. **错误数据格式化**
   - 标准化的错误数据结构
   - 堆栈信息的解析
   - 敏感信息的脱敏
   - 数据压缩

4. **上报策略设计**
   - 立即上报 vs 批量上报
   - 采样与限流
   - 优先级队列
   - 离线存储与补报

5. **上报通道选择**
   - XMLHttpRequest / Fetch
   - navigator.sendBeacon
   - Image Beacon
   - 各通道的优缺点

6. **SourceMap 还原**
   - SourceMap 的原理
   - 服务端还原流程
   - 安全性考虑
   - 还原性能优化

## 4. 写作要求

### 开篇方式
从"为什么需要错误上报"开始：用户不会主动报告问题，线上错误如果不上报就是"黑箱"。引出构建错误上报系统的必要性。

### 结构组织
```
1. 线上错误是一个黑箱 (问题引入)
2. 错误上报系统架构概览 (全局视图)
3. 错误数据的采集与格式化 (数据层)
4. 上报策略与通道选择 (传输层)
5. SourceMap 还原机制 (关键技术)
6. 采样、限流与告警 (运维考量)
7. 开源方案对比 (技术选型)
8. 小结
```
