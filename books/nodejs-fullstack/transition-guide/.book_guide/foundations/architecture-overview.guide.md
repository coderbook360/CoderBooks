# Node.js 架构快速概览

## 章节定位

本章是第一部分的第二章，目标是让读者对 Node.js 的整体架构有一个清晰的认知。不需要深入原理（那是 01-core-principles 的内容），但要建立正确的心智模型。

## 学习目标

读完本章，读者应该能够：

1. 画出 Node.js 的核心架构图（JavaScript 层 → Node.js 绑定层 → libuv/V8）
2. 理解 V8 引擎在 Node.js 中的角色
3. 理解 libuv 是什么以及为什么需要它
4. 知道 Node.js 是"单线程"的含义和局限
5. 理解 Node.js 的核心优势：非阻塞 I/O

## 核心知识点

### 1. 三层架构模型

```
┌─────────────────────────────┐
│     JavaScript 应用代码      │
├─────────────────────────────┤
│   Node.js 标准库 (fs, http)  │
├─────────────────────────────┤
│    Node.js 绑定层 (C++)      │
├──────────────┬──────────────┤
│     V8      │    libuv     │
│  (JS引擎)    │  (异步I/O)   │
└──────────────┴──────────────┘
```

### 2. V8 引擎的角色

- 将 JavaScript 编译为机器码执行
- 管理 JavaScript 内存（垃圾回收）
- 提供 JavaScript 运行时环境
- **重点**：V8 只负责 JavaScript，不负责 I/O

### 3. libuv 的角色

- 提供跨平台的异步 I/O 能力
- 实现事件循环
- 管理线程池（用于无法异步的操作，如 fs）
- 统一 Windows/Linux/macOS 的差异

### 4. 单线程的真相

- **单线程指的是**：JavaScript 代码在单个线程执行
- **多线程的部分**：libuv 线程池、某些 C++ 扩展
- **为什么单线程**：避免多线程同步问题，简化编程模型
- **单线程的代价**：CPU 密集任务会阻塞事件循环

### 5. 非阻塞 I/O 的本质

- I/O 操作委托给操作系统或线程池
- JavaScript 线程不等待 I/O 完成
- I/O 完成后通过回调通知
- 这就是 Node.js 高并发的秘密

## 示例代码片段

```javascript
const fs = require('fs');

console.log('1. 开始读取文件');

// 非阻塞调用：立即返回，不等待
fs.readFile('data.txt', 'utf8', (err, data) => {
  console.log('3. 文件读取完成');
});

console.log('2. 继续执行其他代码');

// 输出顺序：1 → 2 → 3
// 这就是非阻塞 I/O
```
