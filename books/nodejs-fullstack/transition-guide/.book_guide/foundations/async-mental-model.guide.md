# 从单线程理解异步思维模型

## 章节定位

本章是理解 Node.js 编程范式的关键章节。目标是让前端开发者理解为什么 Node.js 采用异步编程模型，以及如何建立正确的异步思维。

## 学习目标

读完本章，读者应该能够：

1. 理解同步阻塞和异步非阻塞的本质区别
2. 建立"别阻塞事件循环"的核心意识
3. 理解异步编程的心智模型：发起 → 继续 → 回调
4. 能够识别代码中的阻塞操作
5. 理解为什么 Node.js 适合 I/O 密集型而非 CPU 密集型任务

## 核心知识点

### 1. 同步 vs 异步的本质

- **同步**：调用后等待结果，代码顺序执行
- **异步**：调用后立即返回，结果稍后通知
- **类比**：餐厅点餐（同步等待 vs 叫号系统）

### 2. 阻塞 vs 非阻塞

- **阻塞**：操作期间线程被占用，无法处理其他任务
- **非阻塞**：操作期间线程可以处理其他任务
- **关键认知**：异步通常是非阻塞的，同步通常是阻塞的

### 3. 事件循环思维

- 事件循环不断检查是否有待处理的事件
- 你的代码是"事件处理器"
- 每个处理器应该快速完成，避免阻塞循环

### 4. 异步编程心智模型

```
发起异步操作
    ↓
立即返回，继续执行后续代码
    ↓
...（事件循环处理其他事件）...
    ↓
操作完成，回调函数被调用
```

### 5. 常见阻塞操作

- `fs.readFileSync` 等同步 API
- CPU 密集计算（大循环、复杂算法）
- `JSON.parse` 大型 JSON
- 正则表达式回溯

### 6. I/O 密集 vs CPU 密集

- **I/O 密集**：大量时间花在等待 I/O（网络、磁盘）→ Node.js 擅长
- **CPU 密集**：大量时间花在计算 → Node.js 不擅长
- **解决方案预告**：Worker Threads、子进程

## 示例代码片段

```javascript
// 阻塞版本：整个服务器在读取文件时无法响应其他请求
const data = fs.readFileSync('large-file.txt');
processData(data);

// 非阻塞版本：读取文件期间可以处理其他请求
fs.readFile('large-file.txt', (err, data) => {
  processData(data);
});
handleOtherRequests(); // 立即执行
```

```javascript
// 阻塞事件循环的反面教材
function heavyComputation() {
  let result = 0;
  for (let i = 0; i < 1e9; i++) { // 10亿次循环
    result += i;
  }
  return result;
}

// 在这个计算期间，服务器完全无法响应任何请求！
```
