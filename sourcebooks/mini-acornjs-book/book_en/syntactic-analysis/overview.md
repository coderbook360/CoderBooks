# Syntactic Analysis Overview: Recursive Descent Method

In the previous chapters, we successfully transformed a stream of raw source code characters into a flat, one-dimensional sequence of Tokens. This is like identifying all the words in an English novel and listing them on a piece of paper. We have the "words," but we don't yet know how they form "sentences" and "paragraphs."

This is the moment when **Syntactic Analysis (Parsing)** takes the stage. Its core task is to transform this flat Token sequence into a tree structure that precisely reflects the logical structure and hierarchical relationships of the code—the **Abstract Syntax Tree (AST)**—according to the grammar rules of the target language.

![Token Flow to AST](https://i.imgur.com/A8gZ3rA.png)  *(From Token Flow to AST)*

## The Leap from One Dimension to Two Dimensions

Imagine the code `if (a) { b; }`. After lexical analysis, we get the following Token sequence:

`[ IF, LPAREN, IDENTIFIER("a"), RPAREN, LBRACE, IDENTIFIER("b"), SEMI, RBRACE ]`

This sequence is linear; it doesn't tell us that `a` is the condition of the `if` statement, and `{ b; }` is its execution body. The goal of syntactic analysis is to "dimensionalize" this inherent, nested logical relationship, constructing a tree structure like this:

```json
{
  "type": "IfStatement",
  "test": {
    "type": "Identifier",
    "name": "a"
  },
  "consequent": {
    "type": "BlockStatement",
    "body": [
      {
        "type": "ExpressionStatement",
        "expression": {
          "type": "Identifier",
          "name": "b"
        }
      }
    ]
  },
  "alternate": null
}
```

This tree clearly shows: this is an `IfStatement` node, its `test` property is an `Identifier` (a), and its `consequent` property is a `BlockStatement`. This is the magic of AST.

So, how do we implement this transformation process? For handwritten parsers, the most popular and intuitive method is **Recursive Descent Parsing**.

## Core Idea: One Rule, One Function

The idea of recursive descent parsing is extremely elegant and concise: **Write a corresponding parsing function for each grammar rule (non-terminal) in the language.**

For example, JavaScript's grammar has syntax rules named `IfStatement`, `WhileStatement`, `Expression`, etc. Therefore, in our parser, there will be corresponding functions like `parseIfStatement()`, `parseWhileStatement()`, `parseExpression()`, etc.

When a parsing function needs a certain sub-syntax structure, it directly calls the corresponding sub-parsing function. This process continues to "descend" until it encounters the most basic elements—**Terminals**—which are the Tokens generated by our lexical analyzer.

Let's experience this process through an example of an `if` statement.

The grammar rule for an `if` statement (simplified version) can be described as follows:

`IfStatement ::= 'if' '(' Expression ')' Statement ('else' Statement)?`

This rule tells us: an `if` statement consists of the `if` keyword, a left parenthesis, an expression, a right parenthesis, a statement, and an optional `else` branch.

According to the recursive descent method, we can write the pseudocode for the `parseIfStatement` function:

```plaintext
function parseIfStatement():
  // 1. Consume the 'if' keyword
  // We expect the current Token to be 'if'. After confirmation, "eat" it and advance the parser to the next Token
  expect(tt._if)

  // 2. Consume '('
  expect(tt.parenL)

  // 3. Call the subordinate parsing function to parse the expression inside the parentheses
  // This is where "recursive descent" is embodied!
  const test = parseExpression()

  // 4. Consume ')'
  expect(tt.parenR)

  // 5. Call the subordinate parsing function to parse the if statement's body
  const consequent = parseStatement()

  // 6. (Optional) Handle the else branch
  let alternate = null
  // If the current Token is 'else', eat it and parse the subsequent statement
  if (eat(tt._else)):
    alternate = parseStatement()

  // 7. All parts are parsed, assemble into an AST node and return
  return new IfStatement(test, consequent, alternate)
```

## Function Call Stack and AST Mapping

The most wonderful aspect of recursive descent parsing is that **the function call stack structure during the parsing process forms a perfect mirror image of the final generated AST structure**.

When parsing `if (a) { b; }`:

1.  The top-level `parseStatement()` is called. It sees that the current Token is `if`, so it decides to call `parseIfStatement()`.
2.  `parseIfStatement()` starts executing. When parsing the conditional expression, it calls `parseExpression()`.
3.  `parseExpression()` finds that the Token is the identifier `a`. It completes its task and returns an `Identifier` node.
4.  After obtaining the `Identifier` node, `parseIfStatement()` continues to parse the body, calling `parseStatement()` again to parse `{ b; }`.

This function call chain `parseStatement -> parseIfStatement -> parseExpression` exactly corresponds to the hierarchical relationship in the AST where the `IfStatement` node contains an `Expression` child node. The syntax structure of the code is naturally constructed through the mutual calls of parsing functions.

## Challenges and Prospects

Of course, recursive descent parsing is not omnipotent. It has a famous "Achilles' heel": **It cannot handle left-recursive** grammar rules.

Rules like `Expression ::= Expression '+' Term` are left-recursive because the left side of the rule (`Expression`) appears directly at the beginning of the right side. If we directly write a `parseExpression()` function for this, it would lead to infinite self-calls: `parseExpression()` -> `parseExpression()` -> ...

In subsequent chapters on expression parsing, we will introduce the powerful **Pratt Parser** to elegantly solve this problem.

## Summary

Syntactic analysis is the core of compiler principles, and recursive descent parsing provides a clear, intuitive path for handwritten parsers. It connects abstract grammar rules with concrete code implementation, using recursive function calls to "sculpt" the one-dimensional Token sequence into a three-dimensional AST.
