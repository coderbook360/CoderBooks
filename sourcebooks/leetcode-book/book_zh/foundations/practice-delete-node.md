# 实战：删除链表中的节点

这是一道非常有趣的"脑筋急转弯"题目。传统的链表删除需要访问前驱节点，但这道题故意刁难你：**只给你待删除的节点本身，不给你链表头**。怎么办？

## 问题描述

有一个单链表，给你其中的一个节点`node`，请你删除它。

**注意**：
- 你**无法访问**链表的头节点
- 题目保证待删除的节点**不是尾节点**
- 链表中每个节点的值都是唯一的

**示例**：
```
输入: 链表 4 → 5 → 1 → 9, node = 5
输出: 4 → 1 → 9
```

乍一看，这似乎是不可能的任务——删除节点需要修改前驱的`next`指针，但我们连前驱在哪都不知道！

## 思路分析

### 传统删除为何行不通

标准的链表删除操作：
```
删除B: A → B → C
需要: A.next = C
```

要执行这个操作，必须能访问到节点A（B的前驱）。但题目只给了我们节点B，无法"回头"找A。

### 换个思路：狸猫换太子

既然无法真正"删除"这个节点，那就让它**伪装成下一个节点**！

核心思路：
1. 把下一个节点的值复制到当前节点
2. 删除下一个节点

```
原链表: 4 → [5] → 1 → 9
         要删除5

步骤1: 复制下一个节点的值
       4 → [1] → 1 → 9
            ↑    
         5变成1了

步骤2: 跳过下一个节点
       4 → [1] → 9
            ↑
         原来的5，现在是1
```

从外部看，效果就像删除了值为5的节点！

## 完整实现

```javascript
/**
 * @param {ListNode} node
 * @return {void} Do not return anything, modify node in-place instead.
 */
function deleteNode(node) {
    // 复制下一个节点的值
    node.val = node.next.val;
    // 删除下一个节点
    node.next = node.next.next;
}
```

就两行代码！

## 图解执行过程

以删除节点5为例：

**初始状态**：
```
4 → 5 → 1 → 9 → null
    ↑
   node(要删除)
```

**步骤1：复制值**
```
node.val = node.next.val
node.val = 1

4 → 1 → 1 → 9 → null
    ↑   ↑
  node  原值1
```

**步骤2：跳过下一节点**
```
node.next = node.next.next
node.next = 9

4 → 1 → 9 → null
    ↑
   node
```

**最终结果**：
```
4 → 1 → 9 → null

从外部看，5被删除了！
```

## 这真的是"删除"吗？

严格来说，这不是真正的删除，而是一种"替身术"：

- **被删除的节点**：实际上是原本的`node.next`（值为1的节点）
- **被修改的节点**：`node`本身，它的值从5变成了1

如果有其他地方保存了指向这些节点的引用，行为可能会出乎意料：

```javascript
// 假设有人保存了对节点5和节点1的引用
const ref5 = node;      // 指向"5"节点
const ref1 = node.next; // 指向"1"节点

deleteNode(node);

console.log(ref5.val); // 1（不是5了！）
console.log(ref1);     // 这个节点被"删除"了，但内存还在
```

在实际面试中，可以提到这个细节，展示你对问题的深入理解。

## 为什么不能删除尾节点？

如果`node`是尾节点：
```
4 → 5 → 1 → 9
            ↑
           node(尾节点)
```

`node.next`是`null`，无法执行：
- `node.val = node.next.val` → 报错：Cannot read property 'val' of null
- 而且即使赋值成功，也无法让前驱节点指向`null`

所以题目限制了**不能是尾节点**。

## 边界情况

| 场景 | 说明 | 处理 |
|------|------|------|
| 只有两个节点，删除第一个 | `1 → 2`，删除1 | 正常工作，变成`2` |
| 待删除节点是尾节点 | 题目保证不会发生 | 无法处理 |
| 节点值重复 | 题目保证值唯一 | 不用担心 |

## 复杂度分析

**时间复杂度：O(1)**
- 只进行了两次指针/值操作

**空间复杂度：O(1)**
- 没有使用额外空间

## 这道题的意义

这道题在实际开发中几乎不会遇到——设计良好的API不会只给你一个节点让你删除。但它考察了：

1. **打破思维定式**：删除节点不一定要"真删除"
2. **对数据结构的理解**：链表节点只是值+指针的组合
3. **创造性思维**：换个角度看问题

在面试中，先表达"传统删除需要前驱节点"的认知，再提出"复制值+删除下一节点"的方案，展示你的问题分析能力。

## 扩展思考

**问题1**：如果需要删除尾节点怎么办？

没有好办法。除非：
- 能够遍历整个链表找到前驱（但题目说无法访问头节点）
- 使用双向链表
- 把尾节点标记为"已删除"（逻辑删除）

**问题2**：如果值可以重复，这个方法还有效吗？

有效！我们删除的是节点，不是值。只要把下一个节点的值复制过来，再跳过它，效果一样。

## 小结

这道题的精髓在于：**当无法删除一个节点时，就让它变成另一个节点，然后删除那个"替身"**。

两行代码背后是对链表本质的深刻理解：
- 链表节点 = 值 + 指向下一节点的指针
- 从外部看，节点的"身份"就是它的值和在链表中的位置

这道题告诉我们：有时候换个思路，看似不可能的问题会有出人意料的简单解法。
