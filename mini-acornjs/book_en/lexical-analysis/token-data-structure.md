# Token Data Structure Design

In the previous chapter, we understood the macro concept of lexical analysis: converting code strings into Token sequences. Now, we must delve into the details to answer a core question: What exactly should this "Token" look like? A robust and efficient Token data structure is the foundation for building a high-performance parser.

## 1. The Problem: Are Simple Strings Sufficient?

The most intuitive idea might be to use a simple JavaScript object to represent a Token, for example:

```javascript
// A Token representing "let"
{
  type: "Keyword",
  value: "let",
  start: 0,
  end: 3
}
```

In the syntax analysis phase, we could determine Token types by comparing `type` strings:

```javascript
if (token.type === "Keyword") {
  // ...handle keyword
}
```

This method is simple and easy to understand, but in a parser with extremely high requirements for performance and robustness, it exposes several serious problems:

1.  **Performance Overhead**: String comparison (`===`) is typically slower than direct object reference comparison. When processing thousands of Tokens, this small performance difference gets amplified.
2.  **Poor Maintainability**: If a developer accidentally makes a typo, like writing `token.type === "Keywod"`, the program won't throw any exception but will fail silently, causing logical errors that are very difficult to track down.
3.  **Weak Information Carrying Capacity**: A simple string `type` cannot carry additional "metadata". For example, when we later implement the Pratt parser, we need to know an operator's "binding power"; or we need to quickly determine if a Token is an assignment operator. Hardcoding this information in the syntax analysis logic would make the code messy.

To solve these problems, we need to design a more professional and powerful Token system.

## 2. Solution: Separating `TokenType` and `Token`

Acorn and many other modern parsers adopt an elegant solution: separating the "definition" of a Token from its "instance".

*   **`TokenType`**: This is a class used to **define** Token types. We create a unique instance of `TokenType` for each lexical unit (such as a keyword, an operator). It can carry all metadata about that type.
*   **`Token`**: This is a class used to represent **instances**. Every Token generated by the lexical analyzer is an instance of the `Token` class, which references a `TokenType` instance as its type and stores its own value and position information.

Let's implement this system by creating two core files: `tokentype.js` and `token.js`.

### 2.1. `tokentype.js`: Defining All Token "Types"

This file is our "dictionary", defining all possible lexical unit types in the language. We'll create a `TokenType` class and use it to instantiate all specific Token types.

```javascript
// src/tokentype.js

export class TokenType {
  constructor(label, options = {}) {
    this.label = label; // String label for debugging and display, e.g., "name", "num"
    this.keyword = options.keyword; // If it's a keyword, store the keyword string here
    // In later chapters, we'll add more metadata here for the Pratt parser, for example:
    // this.binop = options.binop; // Operator precedence
    // this.prefix = options.prefix; // Whether it's a prefix operator
  }
}

// Define an object tt to store all TokenType instances
export const tt = {
  num: new TokenType("num"),
  string: new TokenType("string"),
  name: new TokenType("name"), // Identifier
  eof: new TokenType("eof"),   // End of File

  // Punctuators
  bracketL: new TokenType("["), // [
  bracketR: new TokenType("]"), // ]
  parenL: new TokenType("("),   // (
  parenR: new TokenType(")"),   // )
  dot: new TokenType("."),
  semi: new TokenType(";"),   // ;

  // Operators
  assign: new TokenType("="),
  plus: new TokenType("+"),

  // Keywords
  // Note we use _let because let is a JS reserved word
  _let: new TokenType("let", { keyword: "let" }),
  _if: new TokenType("if", { keyword: "if" }),
};

// To quickly distinguish if an identifier is a keyword, create a mapping from keyword strings to TokenType
export const keywords = new Map([
  ["let", tt._let],
  ["if", tt._if],
]);
```

**Design Analysis**:

*   By using `new TokenType()`, we create a **unique object instance** for each type. Now, in syntax analysis, we can use efficient reference comparison: `token.type === tt.num`.
*   The `TokenType` class acts like an information carrier. The `label` property is for debugging, while the `keyword` property clearly marks which types are keywords.
*   The `keywords` `Map` structure allows us to determine if a read identifier (like `let`) is a keyword in O(1) time complexity.

### 2.2. `token.js`: Defining Token "Instances"

This file defines the actual object structure produced by the lexical analyzer. It's very simple, mainly responsible for storing the specific information of a Token in the code.

```javascript
// src/token.js

export class Token {
  constructor(type, value, start, end, loc) {
    this.type = type;     // A TokenType instance (e.g., tt.num)
    this.value = value;   // Token value (e.g., 10)
    this.start = start;   // Start index in source code
    this.end = end;       // End index in source code
    this.loc = loc;       // A position object containing { line, column }
  }
}
```

Now, when the lexical analyzer parses the number `10`, it creates an instance like this:

```javascript
new Token(tt.num, 10, 10, 12, { ... });
```

## 3. Summary

By separating Token "type definition" (`TokenType`) from "instances" (`Token`), we've built a system that is both efficient and extensible.

*   **Efficiency**: Uses object reference comparison (`===`) instead of string comparison.
*   **Robustness**: Eliminates bugs caused by string typos, since `tt.keywod` would immediately report `undefined`.
*   **Extensibility**: The `TokenType` class can carry any amount of metadata, providing powerful support for subsequent complex syntax analysis (like operator precedence) without modifying the main parser logic.

This data structure is one of the core foundations of mini-acornjs. In the next chapter, we'll use this system to start writing the skeleton of the lexical analyzer.
